# AntiSpamBot

Telegram анти-спам бот на Go. При входе нового участника в группу показывает капчу с 6 цветными кружочками — пользователь должен выбрать кружок указанного цвета за 30 секунд. Неправильный ответ или таймаут → кик (с возможностью перезайти). Три провала подряд → перманентный бан.

Работает в Docker, разворачивается на VDS одной командой, один бинарный файл (~10 МБ), без БД.

## Как это выглядит

Когда в чат заходит новый пользователь, бот отправляет сообщение:

> **Привет, @vasya!**
> Для защиты от спама выбери **красный** кружок за 30 секунд.
> [🟢] [🟡] [🔴] [🔵] [🟣] [🟠]

- Нажал 🔴 → ограничения сняты, сообщение удалено, добро пожаловать.
- Нажал другой цвет / не нажал за 30с → кикается, сообщение удаляется.
- После 3 провалов подряд в этом чате за 24ч — перманентный бан.

Боты (аккаунты с `is_bot=true`), добавленные админами, **не** проверяются.

## Возможности

- Капча с цветными кружками на inline-кнопках
- Настраиваемый таймаут и порог бана
- Работа в нескольких чатах одновременно (с опциональным whitelist'ом)
- Команды `/start` и `/help` в личке с инструкцией
- Структурные JSON-логи (`slog`), уровни `debug|info|warn|error`
- Graceful shutdown по `SIGINT`/`SIGTERM`
- Multi-stage Docker build → итоговый образ ~15 МБ на Alpine
- Запуск от non-root пользователя в контейнере

## Быстрый старт

### 1. Создание бота

1. В Telegram напиши [@BotFather](https://t.me/BotFather) → `/newbot`, придумай имя и username.
2. Сохрани полученный **BOT_TOKEN** вида `123456789:ABC-def...`.
3. У BotFather выключи **Privacy Mode**: `/mybots → <your_bot> → Bot Settings → Group Privacy → Turn off`. Без этого бот не увидит события присоединения.

### 2. Локальный запуск

Нужен Go 1.25+.

```bash
git clone git@github.com:menand/AntiSpamBot.git
cd AntiSpamBot
cp .env.example .env
# впиши BOT_TOKEN в .env
go mod tidy
go run ./cmd/bot
```

### 3. Деплой на VDS через Docker

На сервере нужен только Docker с плагином Compose (Ubuntu/Debian):

```bash
# если Docker ещё не установлен:
curl -fsSL https://get.docker.com | sh

git clone git@github.com:menand/AntiSpamBot.git
cd AntiSpamBot
cp .env.example .env
nano .env   # впиши BOT_TOKEN
docker compose up -d --build
```

Проверить логи:

```bash
docker compose logs -f
```

Остановить:

```bash
docker compose down
```

Обновить до свежей версии:

```bash
git pull
docker compose up -d --build
```

Автостарт при перезагрузке VDS обеспечен `restart: unless-stopped` в `docker-compose.yml`.

## Добавление бота в чат

1. Открой нужную группу → «Управление группой» → «Администраторы» → «Добавить администратора» → найди своего бота.
2. **Обязательные права админа:**
   - ✅ **Блокировать пользователей** (Ban users) — для кика и бана
   - ✅ **Удалять сообщения** (Delete messages) — чтобы убирать сообщение с капчей
3. Остальные права можно выключить.

> ⚠️ **Важно:** без статуса админа Telegram **не присылает** события `chat_member`, и бот вообще не узнаёт о новых участниках. Не «бот сломан», а просто прав нет.

Если используешь `ALLOWED_CHATS` — узнай ID своего чата: перешли любое сообщение из группы в [@userinfobot](https://t.me/userinfobot), он покажет ID вида `-1001234567890`.

## Конфигурация

Все настройки — через переменные окружения (`.env` файл при Docker compose).

| Переменная                | По умолчанию | Описание                                                   |
|---------------------------|--------------|------------------------------------------------------------|
| `BOT_TOKEN`               | **обязат.**  | Токен бота от @BotFather                                    |
| `CAPTCHA_TIMEOUT_SECONDS` | `30`         | Сколько секунд даётся на прохождение капчи                  |
| `MAX_ATTEMPTS`            | `3`          | Через сколько провалов подряд — перманентный бан (вместо кика) |
| `LOG_LEVEL`               | `info`       | `debug` \| `info` \| `warn` \| `error`                      |
| `ALLOWED_CHATS`           | (не задана)  | Chat ID через запятую. Если задан — бот игнорирует другие чаты |

## Команды бота

- `/start` и `/help` — в личке с ботом, показывает краткую инструкцию.
- В группах бот ни на какие команды не реагирует — он работает только на события присоединения и нажатия кнопок капчи.

## Структура проекта

```
cmd/bot/                 entrypoint (main)
internal/config/         парсинг и валидация env-переменных
internal/captcha/        генерация challenge (цвета) + in-memory store активных капч
internal/attempts/       счётчик провалов на (chat_id, user_id) с TTL 24ч
internal/bot/            telego-клиент, хендлеры, действия (restrict/ban/kick/release)
Dockerfile               multi-stage сборка: golang:1.25-alpine → alpine:3.20
docker-compose.yml       одна служба bot, restart: unless-stopped, ротация логов
.env.example             шаблон настроек
CLAUDE.md                заметки по архитектуре для AI-ассистентов
```

## Операционные вопросы

**Логи.** Пишутся в stdout в JSON (`slog`). Docker compose настроен на ротацию: максимум 3 файла по 10 МБ. Посмотреть: `docker compose logs -f` или `docker compose logs --tail=100`.

**Рестарт.** `restart: unless-stopped` — контейнер сам поднимется после падения или ребута VDS. Ручной рестарт: `docker compose restart`.

**Обновления.** `git pull && docker compose up -d --build`. Пересборка занимает ~1 минуту, даунтайм — секунды.

**Бэкапы.** Не нужны — у бота нет персистентного состояния.

**Мониторинг.** Примитивно: `docker compose ps` должен показывать `running`. Можно накинуть внешний healthcheck (например, [UptimeRobot](https://uptimerobot.com/) на публичный эндпоинт), но у бота эндпоинтов нет — проще следить за логами через `docker compose logs` или Loki/Grafana.

## Ограничения

- **Состояние в памяти.** Активные капчи и счётчики провалов хранятся в `map`. После рестарта контейнера:
  - Пользователи, которые были посреди капчи, **останутся с ограничениями** — надо снять вручную или подождать, пока снова сработает `chat_member` (при повторном входе).
  - Счётчики провалов обнуляются — это не страшно, скорее фича.
  - Если критично — нужно добавить SQLite/Redis (интерфейсы `captcha.Store`/`attempts.Tracker` к этому готовы).
- **Один инстанс.** Горизонтальное масштабирование не поддерживается (in-memory state). Для одного бота в нескольких чатах это не проблема: long polling одного инстанса справляется с сотнями чатов.
- **Никакой защиты от продвинутых ботов.** Капча с 6 цветными кружками легко решается ботом с OCR/парсингом текста. Защищает от массовых простых спам-аккаунтов, но не от целенаправленной атаки. Если нужна серьёзная защита — надо усложнять капчу (несколько раундов, картинки и т.д.).

## Траблшутинг

**Бот молчит при входе новых пользователей.**
- Проверь, что бот — **админ** с правами «Банить» и «Удалять сообщения».
- У BotFather отключи **Privacy Mode** для этого бота (`/mybots → Bot Settings → Group Privacy → Turn off`).
- Посмотри логи: `docker compose logs --tail=50`. Если там тихо — Telegram не шлёт апдейты, дело в правах.

**Ошибка `CHAT_ADMIN_REQUIRED`.**
Бот попытался кикнуть / ограничить пользователя без прав. Проверь админские права бота в чате.

**Ошибка `BOT_TOKEN is not set`.**
Нет `.env` файла или в нём пустой `BOT_TOKEN=`. `cp .env.example .env` и впиши токен.

**`CAPTCHA_TIMEOUT_SECONDS must be > 0`.**
В `.env` отрицательное или нулевое значение. Поставь положительное целое (в секундах).

**Пользователь прошёл капчу, но не может писать.**
`release()` не сработал (нет прав или сетевая ошибка). Сними ограничения вручную в настройках чата. В логах будет `release` с err.

**Бот сам кикнулся из чата.**
Возможно, кто-то отозвал админку. Добавь обратно с нужными правами.

**После рестарта контейнера — пользователи «зависли» ограниченными.**
Это известное ограничение MVP (см. раздел [Ограничения](#ограничения)). Сними ограничения вручную либо попроси пользователя выйти и зайти снова.
